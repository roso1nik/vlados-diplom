# Управление сделками с недвижимостью

Проект представляет собой дипломную работу в формате веб-приложения для управления сделками с недвижимостью. Его задача — показать полный жизненный цикл сделки: от заведения объекта и участников до прохождения по статусам, проверки документов, фиксации истории изменений и завершения сделки.

Приложение выполнено как современный frontend-first MVP. В текущей версии основной упор сделан на скорость разработки, наглядность бизнес-процессов и возможность быстро демонстрировать логику предметной области без отдельного серверного слоя. Именно поэтому для хранения данных используется Zustand с `persist`: store сохраняется в `localStorage` и тем самым эмулирует бэкенд. Такой подход позволяет быстро собрать рабочий прототип, проверять сценарии, демонстрировать дипломную часть и не зависеть от готовности API на раннем этапе.

В дальнейшем архитектура рассчитана на естественный переход к полноценной схеме `axios + React Query`, где:

- `axios` будет отвечать за HTTP-взаимодействие с сервером;
- `React Query` будет управлять серверным состоянием, кешированием, синхронизацией и повторными запросами;
- текущий Zustand store можно будет постепенно сократить до локального UI-состояния или демо-режима.

## Назначение проекта

Проект моделирует работу агентства или специалиста по сделкам с недвижимостью. Внутри системы ведутся:

- объекты недвижимости;
- физические лица, выступающие покупателями и продавцами;
- сделки между участниками;
- обязательные и необязательные документы;
- история изменения статусов;
- контроль переходов между этапами сделки.

Основная идея заключается не только в CRUD-операциях, но и в отображении реального делового процесса. Сделка не просто существует как запись в базе, а проходит через понятную цепочку стадий: черновик, переговоры, договор, регистрация, завершение или отмена. Для каждой стадии предусмотрены ограничения, автоматические даты и контроль документов.

## Предметная область

С точки зрения дипломной работы это прикладная информационная система для сопровождения сделок с недвижимостью. Она решает сразу несколько задач:

1. централизует данные об объектах, клиентах и сделках;
2. помогает контролировать последовательность этапов сделки;
3. уменьшает риск ошибок за счет валидации и правил переходов;
4. хранит историю изменений, что важно для аудита и аналитики;
5. позволяет наглядно демонстрировать, как должна работать система сопровождения риэлторского процесса.

Проект ориентирован на типичный сценарий агентства недвижимости, где важно держать в одном интерфейсе сведения об объекте, покупателе, продавце, сумме сделки, задатке, регистрации и пакете документов.

## Основные сущности

### Недвижимость

Сущность недвижимости описывает объект сделки. В проекте поддерживаются:

- адрес;
- тип объекта;
- площадь;
- цена;
- кадастровый номер;
- дополнительные параметры вроде количества комнат, этажа и общего числа этажей;
- текстовое описание;
- даты создания и обновления.

Типы недвижимости включают квартиру, дом, коммерческий объект и землю. Это дает проекту универсальность и делает его ближе к реальному прикладному решению.

### Пользователь

В системе пользователь моделирует физическое лицо, участвующее в сделке. У него хранятся:

- ФИО;
- телефон и email;
- роль в сделке;
- паспортные данные;
- ИНН;
- даты создания и обновления.

Роль может быть покупателем, продавцом или совмещать обе роли. Это позволяет не ограничивать сценарии только одной моделью поведения.

### Сделка

Сделка является центральной сущностью проекта. Она связывает недвижимость, покупателя и продавца, а также хранит:

- сумму сделки;
- задаток;
- финальную сумму;
- текущий статус;
- признак подписания договора;
- даты ключевых этапов;
- номер Росреестра;
- причину отмены;
- автора отмены;
- историю переходов между статусами.

### Документы

Документы нужны для контроля готовности сделки к следующему этапу. В проекте реализован набор обязательных и необязательных документов, привязанных к сделке. Среди них:

- паспорт покупателя;
- паспорт продавца;
- правоустанавливающий документ;
- договор купли-продажи;
- выписка из ЕГРН;
- кадастровый паспорт.

Особенность здесь в том, что система не просто хранит список файлов, а проверяет, готовы ли обязательные документы к переходу сделки в финальный статус.

## Жизненный цикл сделки

Сделка проходит по определенному маршруту статусов:

- `draft` — черновик;
- `negotiation` — переговоры;
- `contract` — подписание договора;
- `registration` — регистрация;
- `completed` — завершена;
- `cancelled` — отменена.

Логика переходов построена так, чтобы имитировать реальный бизнес-процесс:

- нельзя произвольно перепрыгнуть через этапы;
- для перехода в регистрацию требуется подписанный договор;
- для завершения сделки должны быть готовы обязательные документы и указан номер Росреестра;
- при смене статуса фиксируется история изменений;
- при переходах автоматически заполняются связанные даты.

Такой подход делает приложение не просто формой для ввода данных, а системой контроля делового сценария.

## Архитектура проекта

Проект организован по модульному принципу с разделением на слои и предметные области. В структуре используются следующие смысловые уровни:

- `app` — маршруты, layout-слои, локализация и системные страницы Next.js;
- `entities` — доменные сущности и их модели, типы, store и базовая бизнес-логика;
- `features` — пользовательские действия и переиспользуемые сценарии;
- `widgets` — крупные композиционные блоки интерфейса;
- `shared` — общие утилиты, API-клиенты, провайдеры, UI-компоненты, конфиги и стили.

Такой подход помогает держать код проекта структурированным и масштабируемым. Для дипломной работы это особенно полезно, потому что архитектура выглядит не как набор случайных компонентов, а как осознанная система с разделением ответственности.

## Интерфейс и навигация

Главная страница открывает dashboard по сделкам. Внутри интерфейс разделен на несколько ключевых вкладок:

- недвижимость;
- пользователи;
- сделки;
- документы;
- активные сделки и фильтрация по статусам.

Dashboard показывает не только таблицы, но и прикладную логику: редактирование, удаление, переходы между состояниями, отметки документов, отмену сделки и сброс демо-данных. Это помогает быстро продемонстрировать работу системы на защите или в демо-сценарии.

Также в проекте присутствуют отдельные маршруты для авторизации и вспомогательных страниц, а структура локализации построена через `next-intl`.

## Локализация

Проект поддерживает русский и английский языки. Это важно как с точки зрения пользовательского опыта, так и с точки зрения дипломной демонстрации: система выглядит как продукт, рассчитанный на расширение и международное использование.

## Состояние и хранение данных

### Текущее решение

Ключевая особенность проекта — использование Zustand store с `persist` как временной замены бэкенда. Это решение выбрано сознательно по нескольким причинам:

- быстрее собирать MVP без отдельного сервера;
- проще демонстрировать дипломный проект в автономном виде;
- не нужно ждать готовности API для проверки экранов и бизнес-логики;
- данные не теряются после перезагрузки благодаря сохранению в `localStorage`.

Store хранит основные коллекции:

- объекты недвижимости;
- пользователей;
- сделки;
- документы.

Кроме того, в store сосредоточена доменная логика: создание сущностей, обновление, удаление, проверка условий перехода, переходы между статусами, отмена сделки, вычисление прогресса и генерация демо-данных.

### Почему это важно для MVP

Для стадии MVP это наиболее практичный вариант. Он позволяет быстро проверить:

- удобство интерфейса;
- полноту сценариев;
- корректность связей между сущностями;
- логику ограничений;
- качество пользовательского потока.

По сути, `persist` в этом проекте играет роль локального слоя данных, имитирующего бекенд, но без сетевой сложности. Это особенно полезно в дипломной разработке, где на первом месте обычно стоит функциональная демонстрация решения.

### Будущая эволюция архитектуры

Когда проект будет готов к интеграции с сервером, логика может быть перестроена так:

1. `axios` станет транспортным слоем для запросов к API;
2. `React Query` возьмет на себя загрузку и кеширование серверных данных;
3. Zustand останется для локального UI-состояния, временных форм, фильтров или режима демо;
4. бизнес-операции будут перенесены на серверную сторону, а клиент будет только отображать и синхронизировать данные.

Такой путь позволяет безболезненно перейти от прототипа к полноценному production-подобному решению.

Формы в проекте используются для создания и редактирования недвижимости, пользователей и сделок. Они реализуют:

- проверку заполненности ключевых данных;

- пользовательские уведомления об ошибках и успешных действиях.

Это делает интерфейс достаточно реалистичным для прикладной системы, а не просто демонстрационным набором карточек.

## Визуальный слой

Для интерфейса используются современные библиотечные решения, которые позволяют собрать аккуратный и функциональный UI без перегрузки ручной версткой. В проекте виден акцент на:

- понятную иерархию карточек и таблиц;
- быстрые действия через кнопки и модальные окна;

Дополнительно применяются уведомления, индикаторы загрузки и компоненты для более удобной навигации по большим наборам данных.

## Данные по умолчанию

При первом запуске, если локальное хранилище пустое, в систему подгружается демонстрационный набор данных. Он нужен, чтобы:

- сразу показать работу интерфейса;
- избежать пустого состояния на старте;
- ускорить презентацию проекта;
- продемонстрировать все основные типы сущностей без ручного заполнения.

Это особенно удобно для дипломной защиты, потому что можно сразу показать не только пустую оболочку, а полноценный рабочий кейс.

## Серверная интеграция и API-слой

В проекте уже присутствует базовая подготовка к серверному взаимодействию. Создан отдельный `axios`-клиент с поддержкой авторизации и обработкой refresh token-сценария. Это показывает, что архитектура не ограничивается только локальным хранилищем и изначально предполагает дальнейшее развитие в сторону полноценного backend-frontend взаимодействия.

На текущем этапе клиентская часть может работать автономно, но структура уже готова к подключению реального API.

## Структура проекта

Ниже приведена укрупненная схема каталогов и их назначения:

- `src/app` — страницы, layout-и, загрузочные экраны, служебные маршруты;
- `src/entities` — доменные сущности: сделка, недвижимость, пользователь, документы;
- `src/features` — логические пользовательские действия вроде входа, регистрации и подтверждения email;
- `src/widgets` — составные блоки интерфейса: хедер, сайдбар, dashboard, логин-карточки, тема, язык и т.д.;
- `src/shared` — общие утилиты, стили, API, типы, провайдеры и вспомогательные UI-элементы.

Такая декомпозиция улучшает читаемость кода и упрощает масштабирование по мере роста проекта.
